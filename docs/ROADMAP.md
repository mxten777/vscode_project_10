# BidSight 제품 실행 로드맵

이 문서는 BidSight를 시연용 화면이 아니라, 클라이언트가 실제로 필요로 하는 기능 중심의 제품으로 정리하기 위한 단계별 실행 기준입니다.

목표는 단순합니다.

- 입찰 실무자가 반복 작업을 덜 하게 만든다.
- 중요한 공고를 놓치지 않게 만든다.
- 참여 여부를 더 빠르고 더 근거 있게 판단하게 만든다.
- UI/UX 가 기능 나열이 아니라 업무 흐름을 따라가게 만든다.

## 0. 현재 판단

현재 BidSight는 필요한 기능의 씨앗은 갖고 있지만, 제품 전체가 아직 `클라이언트 업무 흐름 중심` 으로 완전히 정렬되지는 않은 상태입니다.

지금 가장 중요한 것은 새 기능 추가보다 아래 4가지를 동시에 만족시키는 것입니다.

1. 클라이언트가 바로 이해하는 문제 정의
2. 검색 → 선별 → 검토 → 추적 → 분석으로 이어지는 일관된 사용자 흐름
3. 화면마다 목적이 분명한 UI/UX 구조
4. 운영과 데이터가 실제 사용 기준으로 안정적인 상태

## 1. 이번 로드맵의 원칙

### 지금 하는 일

- 클라이언트 문제와 핵심 효익을 문서와 화면에서 일치시킨다.
- 자주 쓰는 기능부터 우선순위대로 재정렬한다.
- UI/UX 를 예쁜 카드 나열이 아니라 업무 흐름 중심으로 재구성한다.
- 현재 실제로 동작하는 기능의 사용성을 높인다.
- 운영 리스크는 사용자 경험을 해치지 않는 방향으로 줄인다.

### 지금 하지 않는 일

- AI 기능을 실제보다 앞서 보이게 만드는 일
- 결제 흐름을 제품 핵심보다 먼저 전면에 두는 일
- 데스크톱과 모바일 모두에서 가치가 불분명한 장식성 UI 추가
- 핵심 흐름과 무관한 새 기능 확장

## 2. 단계별 실행 계획

### Stage 1. 클라이언트 가치와 핵심 문제 고정

목표: 이 제품이 누구의 어떤 문제를 해결하는지 흔들리지 않게 만든다.

완료 조건:

- 주요 사용자군이 정리돼 있다.
- 각 사용자군의 핵심 문제 3개 이내가 정리돼 있다.
- 제품의 핵심 효익 3~4개가 고정돼 있다.
- README, 고객용 문서, 랜딩의 중심 메시지가 일치한다.

우선 작업:

- `docs/CLIENT_OVERVIEW.md` 를 기준 문서로 삼아 가치 문장 고정
- `README.md` 와 공개 랜딩에서 같은 톤 유지
- "무엇을 보여줄까" 보다 "무엇을 해결할까" 기준으로 문구 재작성

### Stage 2. 핵심 사용자 흐름 정리

목표: 사용자가 실제 업무에서 밟는 순서대로 제품 흐름을 정리한다.

완료 조건:

- 시작 화면이 명확하다.
- 검색 → 선별 → 상세 검토 → 저장/알림 → 분석 흐름이 끊기지 않는다.
- 각 단계의 입력과 출력이 분명하다.
- 중복되거나 목적이 겹치는 화면 요소가 정리된다.

우선 작업:

- 로그인 전후 핵심 진입 흐름 정의
- 메인 목록에서 상세로 넘어가는 이유 정리
- 상세에서 즐겨찾기/알림/분석으로 이어지는 연결 구조 점검

### Stage 3. 정보 구조와 화면 우선순위 재설계

목표: 메뉴와 화면 구성을 기능 목록이 아니라 업무 우선순위에 맞춘다.

완료 조건:

- 헤더 네비게이션이 사용자 목적 기준으로 읽힌다.
- 로그인 후 첫 화면이 "오늘 봐야 할 것" 중심으로 정리된다.
- 랜딩은 소개 화면이 아니라 문제와 해결 구조를 보여준다.
- 각 페이지의 1차 CTA 가 분명하다.

우선 작업:

- 랜딩 구조 재정비
- 앱 첫 화면 정보 계층 재정렬
- 즐겨찾기/알림/분석 메뉴의 역할 명확화

### Stage 4. 핵심 화면 UI/UX 개편

목표: 자주 쓰는 화면부터 실제 사용성을 높인다.

완료 조건:

- 랜딩에서 핵심 가치가 10초 안에 이해된다.
- 메인 화면에서 검색과 선별이 빠르게 가능하다.
- 상세 화면에서 판단 근거가 우선 노출된다.
- 알림 생성과 즐겨찾기 관리가 짧은 단계로 끝난다.
- 분석 화면이 차트 모음이 아니라 의사결정 보조 도구처럼 읽힌다.

우선 작업:

- 랜딩 단순화 및 가치 중심 문구 정리
- 메인 목록 Hero, 필터, 카드 구조 재설계
- 상세 화면 정보 우선순위 조정
- 알림 규칙 생성 흐름 단순화
- 분석 대시보드 읽는 순서 개선

### Stage 5. 기능 중심 구현 보강

목표: UI 개선이 실제 데이터와 기능 흐름에 맞물리도록 구현을 정리한다.

완료 조건:

- 검색/필터 동작이 일관된다.
- 빈 상태, 오류 상태, 로딩 상태가 자연스럽다.
- 즐겨찾기/알림/분석 기능이 실제 업무 흐름과 맞게 동작한다.
- 핵심 기능이 모바일과 데스크톱에서 모두 사용 가능하다.

우선 작업:

- 검색과 정렬 파라미터 사용성 점검
- 상세 페이지 액션 흐름 점검
- 알림 규칙 CRUD 와 로그 흐름 점검
- 분석 화면 데이터 읽기 부담 완화

### Stage 6. 운영 안정화와 품질 마감

목표: 클라이언트가 실제로 쓸 수 있는 수준으로 안정성과 일관성을 닫는다.

완료 조건:

- 수집/상태 표시가 사용자 신뢰를 깨지 않는다.
- 문구 톤과 시각 언어가 전체적으로 일관된다.
- 주요 흐름 기준 테스트와 점검 기준이 있다.
- 과장된 메시지 없이 현재 가능한 범위가 분명하다.

우선 작업:

- 운영 상태 문구 정리
- 크론/수집 관련 사용자 노출 방식 점검
- 핵심 경로 테스트 보강
- 문서와 화면의 표현 차이 줄이기

## 3. 우선순위 기준

### Priority A. 반드시 먼저 할 것

1. 클라이언트 가치 문장 고정
2. 핵심 사용자 흐름 정리
3. 랜딩과 앱 첫 화면의 정보 구조 재설계
4. 상세/알림/분석의 업무 연결성 강화

### Priority B. 그 다음 할 것

1. 빈 상태와 오류 상태 개선
2. 모바일 사용성 개선
3. 팀 공유와 리포트 해석 흐름 보강

### Priority C. 지금은 미룰 것

1. AI 추천 전면화
2. Stripe 운영 연결 전면 노출
3. 범위가 큰 신규 기능 추가
4. 핵심 흐름과 무관한 구조 리팩터링

## 4. 의사결정 기준

앞으로의 건별 의사결정은 아래 질문으로만 걸러냅니다.

1. 이 작업이 클라이언트의 반복 업무를 줄이는가
2. 이 작업이 검토 우선순위 판단을 더 쉽게 만드는가
3. 이 작업이 공고 누락 위험을 줄이는가
4. 이 작업이 UI/UX 를 더 명확하고 덜 복잡하게 만드는가

넷 중 어느 것도 아니면 지금은 뒤로 미룹니다.

## 5. 바로 다음 실행 순서

1. 클라이언트 가치 문장과 사용자군 정의를 문서 기준으로 고정
2. 현재 주요 화면을 사용자 흐름 기준으로 재분석
3. 랜딩과 로그인 후 첫 화면의 정보 구조를 다시 설계
4. 공고 상세, 즐겨찾기/알림, 분석 화면을 순서대로 개편
5. 마지막에 운영/품질 검증으로 마감

## 6. UI/UX 원칙

- 한 화면에는 한 가지 주목적만 남긴다.
- 정보는 많이 보여주기보다 우선순위대로 보여준다.
- 문구는 기능 설명보다 사용자의 효익을 먼저 말한다.
- CTA 는 많기보다 분명해야 한다.
- 카드 수를 줄이고 계층, 간격, 타이포그래피로 명확성을 만든다.
- 시각적 멋은 장식보다 구조적 정돈에서 나온다.

## 7. 현재 기준 한 줄 결론

BidSight는 지금 `시연 정리` 단계가 아니라, `클라이언트 가치 고정 + 사용자 흐름 재설계 + UI/UX 정돈 + 핵심 기능 사용성 강화` 단계에 있습니다.
