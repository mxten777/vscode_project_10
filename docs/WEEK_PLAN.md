# BidSight 1주 실행 계획

이 문서는 데모 전 1주일 동안 무엇을 어떤 순서로 끝낼지 정리한 실행표입니다.

원칙은 단순합니다.

- 하루마다 완료 기준이 있어야 한다.
- 각 작업은 데모 이해도, 데모 안정성, 핵심 흐름 완성 중 하나에 직접 연결되어야 한다.
- 당일 목표를 못 끝내면 다음 날로 미루기보다 범위를 줄인다.

## 1. 이번 주 목표

이번 주가 끝났을 때 아래 4가지를 만족시키는 것이 목표입니다.

1. 제품 메시지가 문서와 화면에서 일관된다.
2. 데모 시연 순서와 말할 문장이 고정된다.
3. `cron-ingest` 리스크가 통제되거나 시연 경로에서 제거된다.
4. 검색 → 상세 → 즐겨찾기/알림 → 분석 흐름이 시연 기준으로 자연스럽다.

## 2. 이번 주에 하지 않을 일

- AI 기능을 실제 주력 메시지로 끌어올리는 일
- Stripe 운영 연결
- 큰 구조 리팩터링
- 데모와 무관한 새 기능 추가

## 3. 일별 실행 계획

### Day 1. 기준 문서와 메시지 고정

목표:

- 팀이 같은 기준으로 움직일 수 있도록 문서 기준을 확정한다.

해야 할 일:

- `docs/ROADMAP.md` 기준으로 우선순위 고정
- `docs/DEMO_SCRIPT.md` 검토
- README, 랜딩, 고객용 소개 문구 최종 톤 점검

완료 조건:

- 한 줄 소개가 더 이상 흔들리지 않는다.
- "무엇을 보여주고 무엇을 안 보여줄지" 문서로 합의된다.

### Day 2. 데모 경로 화면 정리

목표:

- 시연에서 처음 보는 사람이 바로 이해할 수 있게 핵심 화면 문구를 정리한다.

해야 할 일:

- 랜딩 핵심 문구 보강
- 앱 첫 진입 또는 주요 CTA 점검
- 필요 없는 과장 문구 제거

완료 조건:

- 공개 랜딩과 데모 진입 화면이 같은 메시지를 말한다.
- 시연자가 설명을 덜 해도 흐름이 보인다.

### Day 3. 운영 리스크 결정

목표:

- `cron-ingest` 문제를 더 이상 미뤄두지 않고 방향을 확정한다.

해야 할 일:

- `collect-bid-awards` 경량화 가능성 확인
- 필요 시 `cron-ingest` 에서 분리하는 방안 선택
- 시연 중 수집 상태 질문에 대한 기준 문구 정리

완료 조건:

- `cron-ingest` 대응 방안이 코드 또는 운영 절차로 확정된다.
- 시연 중 타임아웃 리스크 설명이 아니라 대응 상태를 말할 수 있다.

현재 상태:

- 완료. `collect-bid-awards` 경량 모드 추가 후 프로덕션 `cron-ingest` 성공 확인
- 다음 판단 과제는 데모 이후 전체 award 수집을 별도 배치로 분리할지 여부

### Day 4. 핵심 사용자 흐름 점검

목표:

- 검색 → 상세 → 즐겨찾기/알림 → 분석 흐름에서 끊기는 부분을 줄인다.

해야 할 일:

- 데모 기준 대표 공고 선택
- 검색 키워드와 예시 시나리오 고정
- 빈 상태나 전환 문구 중 어색한 부분 수정

완료 조건:

- 시연 리허설을 했을 때 흐름이 끊기지 않는다.

현재 상태:

- 대표 검색 키워드와 공고 예시는 `docs/DEMO_SCENARIOS.md` 에 고정
- 다음은 실제 리허설에서 이 흐름이 매끄러운지만 확인하면 됨

### Day 5. 리허설 1차

목표:

- 실제 발표 흐름으로 한 번 끝까지 돌려본다.

해야 할 일:

- `docs/DEMO_SCRIPT.md` 기준 리허설
- 걸리는 화면, 긴 설명이 필요한 화면 체크
- 장애 시 우회 경로 확인

완료 조건:

- 1차 리허설에서 막히는 포인트 목록이 나온다.
- 수정해야 할 항목이 우선순위대로 정리된다.

실행 기준 문서:

- `docs/DEMO_SCRIPT.md`
- `docs/DEMO_SCENARIOS.md`
- `docs/REHEARSAL_CHECKLIST.md`

### Day 6. 보정 작업

목표:

- 1차 리허설에서 드러난 문제만 좁게 수정한다.

해야 할 일:

- 문구 수정
- 시연 순서 미세 조정
- 필요한 경우 운영 체크리스트 보완

완료 조건:

- 리허설에서 나온 핵심 문제들이 정리된다.
- 데모 스크립트가 실제 흐름과 맞아떨어진다.

### Day 7. 최종 리허설과 잠금

목표:

- 더 이상 범위를 넓히지 않고 최종 버전을 잠근다.

해야 할 일:

- 최종 리허설
- 계정, 데이터, 경로, 우회 문구 최종 확인
- 발표 직전 체크리스트 고정

완료 조건:

- 누가 시연해도 같은 순서로 진행 가능하다.
- 데모 직전 새로운 기능 추가를 중단한다.

## 4. 매일 끝날 때 확인할 질문

1. 오늘 작업이 데모 메시지를 더 명확하게 만들었는가
2. 오늘 작업이 데모 리스크를 실제로 낮췄는가
3. 오늘 작업 결과가 문서나 코드로 남았는가
4. 내일의 가장 중요한 한 가지가 정해졌는가

## 5. 이번 주 핵심 리스크

### 가장 큰 리스크

- 전체 award 범위를 한 번에 돌릴 경우 다시 실행 시간이 길어질 수 있는 문제

### 운영 리스크

- 시연 중 최신 수집 상태 질문이 들어올 때 설명이 길어질 수 있음

### 메시지 리스크

- AI, 결제, 확장 기능이 실제보다 앞에 보이면 신뢰가 떨어질 수 있음

## 6. 이번 주 성공 기준

이번 주가 끝났을 때 아래를 만족하면 성공입니다.

- 문서 기준이 하나로 정리돼 있다.
- 시연 스크립트가 고정돼 있다.
- 운영 리스크 대응안이 확정돼 있다.
- 시연 흐름이 한 번에 설명 가능하다.
