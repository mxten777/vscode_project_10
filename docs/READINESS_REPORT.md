## 2. 코드는 있지만 실제로 작동하지 않는 기능

| 기능 | 이유 | 필요 조치 |
|------|------|----------|
| **AI 투찰가 추천** | `AI_SERVICE_URL` 미설정 → 100% 503 오류 | Railway bid-ai-service 배포 |
| **벡터 유사 공고** | 동일 이유. `title_embedding` 전부 NULL | 위와 동일 |
| **Stripe 결제** | `STRIPE_SECRET_KEY` 미설정 | Stripe 대시보드 설정 |
| **Auto 임베딩 배치** | `AI_SERVICE_URL` 없어서 Cron 실행해도 즉시 503 | Railway 배포 후 자동 해결 |

> AI 추천과 Stripe는 코드 경로는 있으나 운영 연결이 닫히지 않은 영역입니다. 코어 사용자 흐름과 혼동하지 않는 편이 맞습니다.

## 3. 구조적 문제

### 3-0. 배포 기준 단일화는 아직 필요

- `DEPLOYMENT_GUIDE.md`와 `supabase/schema.sql`만으로는 현재 서비스 스키마를 완전히 재현할 수 없음
- 실제 AI/분석/구독 기능은 `supabase/migrations/*.sql` 기준으로 확장됨
- 신규 환경 구축 시 문서대로 배포하면 AI 학습, 분석 캐시, 수집 로그 등 일부 기능이 누락될 수 있음
- 즉, 신규 환경 재현성 측면에서는 **기능 추가보다 배포 기준 단일화** 가 여전히 우선입니다.

### 3-1. Vercel Hobby Cron 제한

- 현재 `vercel.json` 은 **상위 Cron 2개** (`cron-ingest`, `cron-maintenance`) 기준으로 정리됨
- `029_expand_collection_logs_for_cron.sql` 적용으로 maintenance 작업 로그(`alerts`, `participants`, `analysis_rebuild`, `cleanup`) 기록 제약은 운영 DB에 반영됨
- 운영 검증 결과 `cron-maintenance` 는 프로덕션에서 실제 호출되어 `process-alerts`, `rebuild-analysis`, `collect-participants` 성공 로그가 `collection_logs` 에 남는 것까지 확인됨
- `rebuild-analysis` 는 초기 검증에서 `UPDATE requires a WHERE clause` 로 실패했으나, `030_fix_sync_agency_counts_safeupdate.sql` 적용 후 정상 복구됨
- `cron-ingest` 는 초기에는 `collect-bid-awards` 장시간 실행 때문에 `FUNCTION_INVOCATION_TIMEOUT` 이 발생했으나, 현재는 `collect-bid-awards?lookbackDays=2&maxPages=1&maxItems=25` 경량 모드로 호출하도록 조정됨
- 최신 프로덕션 검증에서 `cron-ingest` 는 약 23초, 경량 `collect-bid-awards` 직접 호출은 약 7.6초 내에 성공 응답을 반환함

| Cron | 중요도 | 현재 상태 |
|------|--------|----------|
| poll-tenders | 🔴 핵심 | 실행 중 (확인됨) |
| collect-bid-awards | 🔴 핵심 | 경량 모드(`lookbackDays=2,maxPages=1,maxItems=25`) 프로덕션 성공 확인 |
| cron-ingest | 🔴 핵심 | 경량 awards 모드 기준 프로덕션 성공 확인 |
| process-alerts | 🟠 중요 | 프로덕션 `collection_logs` 성공 기록 확인 완료 |
| collect-participants | 🟠 중요 | 프로덕션 `collection_logs` 성공 기록 확인 완료 |
| rebuild-analysis | 🟡 보통 | `030` 적용 후 프로덕션 성공 기록 확인 완료 |
| cleanup | 🟢 낮음 | 코드상 `collection_logs` 기록 연결됨, 일요일 실행 또는 수동 확인 필요 |
| embed-batch | 🟢 낮음 | AI Service 없어서 무용 |

### 3-2. 낙찰 데이터 부족

- 실제 award 데이터: **~280건** (최근 1개월)
- `startMonthsAgo=1` 백필: **6건**, `=2`: **0건** → 과거 데이터 사실상 없음
- AI 예측 모델 배포해도 **학습 데이터 부족** 상태
- 통계적으로 의미있는 추천을 위해서는 최소 1,000건 이상 권장

### 3-3. 테스트는 실용 수준이지만 빈도가 높은 경로는 더 넓힐 수 있음

- 핵심 API 라우트, 스키마, cron/job 경로는 Vitest 회귀가 계속 넓어지는 중입니다.
- 공개 페이지와 로그인 후 핵심 흐름은 Playwright smoke/E2E로 보호됩니다.
- 다만 tenders 검색, reports summary 같은 사용 빈도 높은 API는 추가 회귀를 더 붙일 여지가 있습니다.

```

████████████████████░░ 90%
```

## 6. 결론

현재 BidSight는 **핵심 업무 흐름이 연결된 운영 가능 코어 제품**입니다.

- **강점**: 검색, 저장 검색, 즐겨찾기, 알림, 리포트, cron 수집이 실제 사용자 흐름으로 연결됨
- **약점**: AI 미연결, Stripe 미운영, 어드민 운영 UI 부재, 추가 알림 채널 부재, 배포 기준 문서 단일화 필요

**다음 단계는 현재 코어 흐름을 흔들지 않은 채 운영도구, 문서 기준, 알림 채널, 수익화 연결을 닫는 것입니다. 즉 새 기능 확장보다 남은 운영 마감 항목을 끝내는 편이 현재 우선순위에 맞습니다.**
