# BidSight 리허설 체크리스트

이 문서는 Day 5 리허설 1차와 Day 7 최종 리허설에서 그대로 사용하는 점검표입니다.

기준 문서는 아래 2개입니다.

- `docs/DEMO_SCRIPT.md`
- `docs/DEMO_SCENARIOS.md`

## 1. 리허설 목표

이번 리허설의 목표는 3가지입니다.

1. 시연 흐름이 끊기지 않는지 확인한다.
2. 설명이 길어지는 구간을 찾는다.
3. 장애나 질문이 들어왔을 때 우회 문구가 자연스러운지 확인한다.

## 진행 방식 원칙

리허설도 실제 시연과 동일하게 `탑다운` 으로 진행합니다.

확인할 점:

- 랜딩 → 검색 → 상세 → 알림 → 분석 순서가 유지되는가
- 중간에 아직 설명하지 않은 화면으로 점프하지 않는가
- 각 단계가 다음 단계의 이유를 만들어 주는가

## 2. 리허설 전 준비

### 환경 확인

- 프로덕션 URL 접속 가능
- 로그인 계정 준비 완료
- 네트워크 상태 확인
- 브라우저 탭 정리 완료
- 불필요한 알림/팝업 차단

### 시연 데이터 확인

- 검색 키워드: `플랫폼`
- 상세 공고: `경기도 공공빅데이터 플랫폼 구축 1단계`
- 알림 설명용 키워드: `AI`
- 백업 키워드: `AI`, `용역`

### 발표 흐름 확인

- 한 줄 소개 기억
- 각 단계별 핵심 메시지 기억
- AI/결제 관련 과장 금지 문구 숙지
- 장애 시 우회 문구 준비

## 3. 리허설 진행 체크

### Step 1. 문제 제기

- 20초 안에 문제를 설명했는가
- 서비스 대상이 분명하게 들리는가
- 문제 설명이 기능 소개보다 먼저 나오는가

### Step 2. 랜딩 설명

- 랜딩에서 서비스 가치가 바로 보이는가
- "공고를 모은다 / 고른다 / 놓치지 않게 한다 / 판단 근거를 준다" 순서가 유지되는가
- 설명이 추상적이지 않은가

### Step 3. 검색과 선별

- `플랫폼` 검색 결과가 안정적으로 보이는가
- 여러 OPEN 공고 중 무엇을 먼저 볼지 자연스럽게 설명되는가
- 검색 결과 수보다 선별 속도를 강조하고 있는가

### Step 4. 공고 상세와 판단 근거

- 결과 공고 상세로 자연스럽게 전환되는가
- 낙찰 정보가 실제로 보이는가
- "판단 근거" 라는 표현이 분명하게 전달되는가

### Step 5. 즐겨찾기와 알림

- `AI` 키워드 기반 관심군 설명이 자연스러운가
- 즐겨찾기와 알림의 차이가 짧게 설명되는가
- "좋은 공고를 찾는 것" 보다 "놓치지 않는 구조" 가 강조되는가

### Step 6. 분석 대시보드 / 리포트

- 분석 화면으로 넘어가는 전환이 갑작스럽지 않은가
- 전체 흐름 속 의사결정 지원이라는 메시지가 유지되는가
- 화면 설명이 기능 나열로 무너지지 않는가

### Step 7. 마무리

- 첫 문장과 마지막 문장이 같은 메시지로 닫히는가
- 서비스 가치가 한 문장으로 다시 정리되는가
- 시간이 과도하게 늘어나지 않는가

## 4. 시간 체크

권장 시간은 아래 기준을 넘기지 않는 편이 좋습니다.

| 구간 | 권장 시간 |
|------|-----------|
| 문제 제기 + 랜딩 | 1분 |
| 검색과 선별 | 1분 |
| 상세와 판단 근거 | 1분 |
| 즐겨찾기/알림 | 1분 |
| 분석/리포트 + 마무리 | 1분 |
| 총합 | 5분 내외 |

## 5. 리허설 중 기록할 항목

아래 항목이 보이면 바로 기록합니다.

- 설명이 20초 이상 길어지는 화면
- 클릭 경로가 헷갈리는 화면
- 데이터가 약하거나 덜 설득력 있는 화면
- 질문이 들어오면 흔들릴 만한 구간
- 장애 시 백업 시나리오가 필요한 구간
